# ObfusCode-Bench 困难任务设计

**设计目标**: 为Semantic Momentum实验设计足够困难的任务，使得：

1. 混淆后的代码需要多步迭代才能修复
2. 容易产生震荡（修复A引入B，修复B又破坏A）
3. 需要维护全局一致性，不能局部修复
4. 测试用例提供的梯度信号较弱

**设计原则**:

- 任务本身具有中等以上难度（LeetCode Medium-Hard）
- 多个相互依赖的组件
- 边界条件复杂，容易遗漏
- 状态管理关键
- 混淆后语义完全丢失

---

### 参考实现（将被混淆）

```python
def evaluate_expression(expr: str) -> float:
    """表达式计算器"""
    expr = expr.replace(" ", "")
  
    def parse_number(s, pos):
        """解析数字"""
        start = pos
        if pos < len(s) and s[pos] in '+-':
            pos += 1
        while pos < len(s) and (s[pos].isdigit() or s[pos] == '.'):
            pos += 1
        if start == pos:
            raise ValueError("Expected number")
        return float(s[start:pos]), pos
  
    def parse_factor(s, pos):
        """解析因子（数字或括号表达式）"""
        if pos >= len(s):
            raise ValueError("Unexpected end")
    
        if s[pos] == '(':
            pos += 1
            val, pos = parse_expr(s, pos)
            if pos >= len(s) or s[pos] != ')':
                raise ValueError("Mismatched parentheses")
            return val, pos + 1
        else:
            return parse_number(s, pos)
  
    def parse_term(s, pos):
        """解析项（处理*和/）"""
        left, pos = parse_factor(s, pos)
    
        while pos < len(s) and s[pos] in '*/':
            op = s[pos]
            pos += 1
            right, pos = parse_factor(s, pos)
            if op == '*':
                left = left * right
            else:
                if right == 0:
                    raise ValueError("Division by zero")
                left = left / right
    
        return left, pos
  
    def parse_expr(s, pos):
        """解析表达式（处理+和-）"""
        left, pos = parse_term(s, pos)
    
        while pos < len(s) and s[pos] in '+-':
            op = s[pos]
            pos += 1
            right, pos = parse_term(s, pos)
            if op == '+':
                left = left + right
            else:
                left = left - right
    
        return left, pos
  
    try:
        result, pos = parse_expr(expr, 0)
        if pos != len(expr):
            raise ValueError("Invalid expression")
        return result
    except Exception as e:
        raise ValueError(f"Invalid expression: {str(e)}")
```

### 为什么这个任务适合

1. **多层递归**: `parse_expr` → `parse_term` → `parse_factor` → `parse_expr`，混淆后很难理解
2. **状态传递**: `pos` 位置指针需要正确传递，容易出错
3. **优先级处理**: 必须正确实现运算符优先级，局部修复会破坏全局
4. **边界条件多**: 括号匹配、除零、非法输入等
5. **混淆效果极佳**: 变量名如 `v_1`, `v_2` 完全无法理解递归逻辑

### 参考实现

```python
def max_meetings(intervals: list[tuple[int, int]]) -> int:
    """区间调度问题 - 贪心算法"""
    if not intervals:
        return 0
  
    # 关键：按结束时间排序（不是开始时间！）
    sorted_intervals = sorted(intervals, key=lambda x: x[1])
  
    count = 1
    last_end = sorted_intervals[0][1]
  
    for start, end in sorted_intervals[1:]:
        # 如果当前会议的开始时间 >= 上一个会议的结束时间
        if start >= last_end:
            count += 1
            last_end = end
  
    return count
```

### 为什么这个任务适合

1. **算法理解**: 必须理解为什么按结束时间排序，混淆后很难推理
2. **边界条件**: `start >= last_end` vs `start > last_end`，容易出错
3. **反直觉**: 不是选最短的会议，也不是按开始时间排序
4. **测试用例陷阱**: 多个测试用例专门测试错误的贪心策略

### 参考实现（DP解法）

```python
def longest_increasing_subsequence(nums: list[int]) -> tuple[int, list[int]]:
    """最长递增子序列 - 动态规划"""
    if not nums:
        return (0, [])
  
    n = len(nums)
    # dp[i] 表示以 nums[i] 结尾的最长递增子序列长度
    dp = [1] * n
    # parent[i] 记录 nums[i] 在LIS中的前一个元素的索引
    parent = [-1] * n
  
    # 填充DP表
    for i in range(1, n):
        for j in range(i):
            if nums[j] < nums[i] and dp[j] + 1 > dp[i]:
                dp[i] = dp[j] + 1
                parent[i] = j
  
    # 找到最大长度及其位置
    max_length = max(dp)
    max_idx = dp.index(max_length)
  
    # 回溯构造序列
    sequence = []
    idx = max_idx
    while idx != -1:
        sequence.append(nums[idx])
        idx = parent[idx]
  
    sequence.reverse()
    return (max_length, sequence)
```

### 为什么这个任务适合

1. **DP理解**: 需要理解状态定义和转移方程
2. **双重循环**: 混淆后很难理解嵌套循环的逻辑
3. **回溯构造**: 需要正确使用parent数组回溯
4. **多种解法**: 容易写出错误的贪心或其他方法

```python
def word_search(board: list[list[str]], word: str) -> bool:
    """单词搜索 - DFS回溯"""
    if not board or not board[0] or not word:
        return not word
  
    rows, cols = len(board), len(board[0])
  
    def dfs(r, c, idx):
        """DFS搜索"""
        # 找到完整单词
        if idx == len(word):
            return True
    
        # 边界检查
        if r < 0 or r >= rows or c < 0 or c >= cols:
            return False
    
        # 字符不匹配或已访问
        if board[r][c] != word[idx]:
            return False
    
        # 标记为已访问
        temp = board[r][c]
        board[r][c] = '#'
    
        # 四个方向搜索
        found = (dfs(r+1, c, idx+1) or
                 dfs(r-1, c, idx+1) or
                 dfs(r, c+1, idx+1) or
                 dfs(r, c-1, idx+1))
    
        # 回溯：恢复状态
        board[r][c] = temp
    
        return found
  
    # 从每个位置开始尝试
    for i in range(rows):
        for j in range(cols):
            if dfs(i, j, 0):
                return True
  
    return False
```

### 为什么这个任务适合

1. **递归回溯**: 需要正确实现DFS和状态回溯
2. **状态管理**: 必须标记访问并正确恢复
3. **边界条件**: 多个边界检查，容易遗漏
4. **四方向搜索**: 混淆后很难理解方向逻辑
5. **容易出错**: 忘记回溯、重复使用单元格等

def serialize(root: TreeNode) -> str:
    """序列化二叉树 - 前序遍历"""
    if root is None:
        return "null"
  
    result = []
  
    def dfs(node):
        if node is None:
            result.append("null")
            return
    
        result.append(str(node.val))
        dfs(node.left)
        dfs(node.right)
  
    dfs(root)
    return ",".join(result)

def deserialize(data: str) -> TreeNode:
    """反序列化二叉树"""
    if data == "null":
        return None
  
    values = data.split(",")
    index = [0]  # 使用列表保持引用
  
    def dfs():
        if index[0] >= len(values):
            return None
    
        val = values[index[0]]
        index[0] += 1
    
        if val == "null":
            return None
    
        node = TreeNode(int(val))
        node.left = dfs()
        node.right = dfs()
    
        return node
  
    return dfs()
```

from collections import deque

def max_sliding_window(nums: list[int], k: int) -> list[int]:
    """滑动窗口最大值 - 单调队列"""
    if not nums or k == 0:
        return []
  
    # 单调递减队列，存储索引
    dq = deque()
    result = []
  
    for i in range(len(nums)):
        # 移除窗口外的元素
        if dq and dq[0] < i - k + 1:
            dq.popleft()
    
        # 维护单调性：移除所有小于当前元素的
        while dq and nums[dq[-1]] < nums[i]:
            dq.pop()
    
        dq.append(i)
    
        # 窗口形成后记录最大值
        if i >= k - 1:
            result.append(nums[dq[0]])
  
    return result
```

### 为什么这个任务适合

1. **数据结构**: 需要理解单调队列
2. **索引管理**: 队列存储索引而非值
3. **维护逻辑**: while循环的条件容易出错
4. **边界处理**: 窗口形成时机

## 混淆策略建议

### Level 0: 干净代码（基准）

保持原样，用于验证ground truth

### Level 1: 基础混淆

- 变量名: `expr` → `v_1`, `pos` → `v_2`
- 函数名: `parse_expr` → `f_1`, `parse_term` → `f_2`
- 保留结构和注释

- 视觉混淆: `expr` → `IllI`, `pos` → `IlIl`
- 移除所有注释
- 移除所有docstring
- 函数名: `parse_expr` → `fn1Il`

### Level 3: 极限混淆 + 注入错误

- Level 2的所有混淆
- 注入1-2个逻辑错误（如边界条件错误）
- 强制需要多步修复

### 假设（基于任务1）

**Level 0 (干净代码)**:

- Baseline: 0步解决（代码已正确）
- Momentum: 0步解决

**Level 1 (基础混淆)**:

- Baseline: 2-3步，可能震荡（修复括号破坏优先级）
- Momentum: 1-2步，动量抑制震荡

- Baseline: 5-8步，严重震荡（OI > 2.0）
- Momentum: 3-5步，轻微震荡（OI < 1.0）

**Level 3 (极限混淆+错误)**:

- Baseline: 10+步或失败，极度震荡（OI > 5.0）
- Momentum: 5-8步，中度震荡（OI < 2.0）

### 成功指标

如果实验显示：

1. Level 2+时，Baseline的OI显著高于Momentum
2. Momentum的收敛步数少于Baseline
3. 在Level 3时，Baseline失败但Momentum成功

则论文的核心假设得到验证。

## 七、外部数据集与现有架构分析（2026-06-23 补充）

### 7.1 当前环境覆盖情况

现有 5 个环境的覆盖梯度：

```
convex(6) → lru(8) → graph(8) → discount(15) → saddle(26)
纯函数    → 状态结构 → 图算法  → 多规则交互   → lexer↔parser 强耦合
```

覆盖了从简单到复杂的合理梯度，**作为 SMO 方法论的内部验证集已经充分**。

### 7.2 单文件任务是否偏简单？

**对 SMO 核心验证来说够用，但存在三个盲区。**

关键区分：SMO 研究的不是"能不能修好 bug"，而是"动量是否减少振荡"。单文件内耦合（如 saddle 的 lexer↔parser）已经能产生：

```
修 A → 坏 B → 修 B → 坏 A 的振荡循环 ← 这正是 SMO 要解决的问题
```

但以下三个场景单文件任务**无法覆盖**：

| 盲区 | 为什么重要 | 对 SMO 的潜在影响 |
|---|---|---|
| **跨文件依赖** | 真实项目中 bug fix 经常涉及 2-5 个文件 | 动量向量在更大 action 空间中是否仍然有效？ |
| **上下文窗口压力** | 多文件 → 更长的 observation | `_extract_gradient()` 的 LLM 分析质量是否下降？ |
| **修复链长度** | 单文件 bug 通常 1-3 步修好 | SMO 的 EMA 在 10+ 步长链中是否漂移？ |

**结论**：单文件任务对验证 SMO 的核心机制（EMA 动量抗振荡）是充分的。如果要声称"适用于真实开发场景"，需要多文件任务作为补充。

## 八、从验证 SMO 角度出发的补充建议（2026-06-23 补充）

### 8.1 补充"长链"环境（P3，可选）

**问题**：当前所有环境的 `max_steps` 默认 20，但实际 bug 通常 3-5 步就修好了。SMO 的 EMA 动量在长链中的漂移特性没有被测试。

**做法**：在 saddle 环境中引入更深层的混淆（同时破坏 lexer + parser + 两个辅助函数），迫使 agent 需要 10+ 步才能修复。

**预期收益**：展示 SMO 在长链优化中的稳定性优势。

### 8.2 补充"多文件"环境（P2，锦上添花）

**问题**：所有环境都是单文件，无法验证 SMO 在跨文件场景下的有效性。

**最低成本方案**（不需要外部数据集）：

```
environments/env_multi_file.py
├── models.py      ← 数据模型（简单 class）
├── service.py     ← 业务逻辑（import models）
└── tests          ← 跨文件集成测试
```

大约 150-200 行代码，可以回答"SMO 在跨文件场景是否有效"的核心问题。

### 8.3 引入标准化评估指标（P0，必须做）

当前 `metrics.py` 有 `oscillation_index` 和 `ast_distance`，但缺少与外部 benchmark 对标的指标：

| 指标 | 用途 | 实现复杂度 | 状态 |
|---|---|---|---|
| **Pass@k** | 标准代码修复指标，k 次尝试至少一次通过 | 低 | 未实现 |
| **步数-通过率曲线** | 展示 SMO vs baseline 的收敛速度差异 | 低 | 可从 trajectory 数据生成 |
| **Fix 稳定性** | 修复后是否回退（oscillation 的量化） | — | 已有 `oscillation_index` |

### 8.4 消融实验设计（P0，必须做）

论文审稿人会要求的消融维度：

```
核心对比：
  SMO (β=0.9) vs Baseline (β=0)    ← 主实验，已有

需要补充的消融：
  1. β 敏感性：β ∈ {0.5, 0.7, 0.9, 0.95}    ← 动量系数的影响
  2. 梯度提取质量：有/无 gradient extraction   ← SMO 3 步中哪步最关键
  3. 动量注入方式：system prompt vs 额外 token  ← 实现方式的影响
```

### 8.5 QuixBugs 适配（P1，强烈建议）

**做法**：写一个 `QuixBugsEnvironment` adapter（约 100 行代码），将 `json_testcases` 包装成 `SGDEvaluator` 的 `(input, expected)` 格式。

**收益**：
- 低成本高回报——100 行代码即可获得"在标准 benchmark 上验证"的结果
- 显著提升论文可信度——审稿人认可的标准数据集
- 可以直接复用现有的 `SGDEvaluator` 和 `LoopRunner` 基础设施

**适配要点**：
1. 读取 `python_programs/` 中的 buggy code 作为初始代码
2. 读取 `json_testcases/` 中的测试用例，转换为 `(input, expected)` 元组
3. 函数名作为 `entry_point`（如 `quicksort`）
4. 需要处理部分测试用例格式不一致的问题（有的是 nested list，有的是 flat）
